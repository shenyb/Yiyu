# 医语（Yiyu）项目问题排查报告

> 排查日期：2025-07-10  
> 排查范围：后端 Java（Spring Boot 3 + JDK 25）、前端 Vue 3、全流程功能

---

## 一、已确认的严重问题（影响核心可用性）

### 🔴 P1-01：DeepSeek API 超时导致前端"没反应"

**现象**：用户操作后界面卡住无反馈，等很久才报错或无报错。

**根因**：
- `DeepSeekService` 的 `HttpClient` 只设了 `connectTimeout(30s)`，**没有设 readTimeout / responseTimeout**
- `HttpRequest.timeout(Duration.ofSeconds(60))` 虽然设了 60s，但 DeepSeek 生成医学大纲+JSON 的耗时经常超过 60s（尤其是长主题+参考文档时）
- 前端 `axios` timeout 设了 120s，但后端 60s 就抛异常了 → **后端先超时，前端还在傻等**
- 前端没有 loading 动画（只有文字"⏳"），用户以为卡死

**影响范围**：PPT 大纲生成、课题调研大纲、报告生成、文档问答 — 所有需要调用 AI 的接口

**建议修复**：
1. 后端 HttpRequest timeout 调整到 180s，或改为可配置
2. 前端增加真正的 loading spinner + 预估等待提示
3. 增加重试机制（网络抖动时自动重试 1 次）

---

### 🔴 P1-02：PPT 大纲和调研大纲 — 医学内容无出处/引用

**现象**：生成的 PPT 大纲和调研报告内容，每个观点没有任何文献出处，对于医学用途这是致命问题。

**根因**：
- `PptService.generateOutline()` 的 prompt 只要求"内容专业、准确"，**没有要求 AI 标注出处**
- `ResearchController.generateOutline()` 的 prompt 也没要求引用文献
- DeepSeek 本身会"幻觉"编造内容，对于医学领域这很危险

**建议修复**：
1. PPT 大纲 prompt 增加：每个要点必须附带 `[出处: 作者/年份/期刊]` 格式的引用
2. 调研大纲 prompt 增加：每个 subPoint 必须注明文献来源
3. JSON 结构扩展：每个 item 增加 `reference` 字段
4. 增加 system prompt 校验：如果 AI 没给出处，提示用户"此观点缺乏文献支持，请核实"

---

### 🔴 P1-03：DeepSeekService.parseResponse() 解析极易失败

**现象**：AI 返回正常但前端报"AI 回复解析失败"或 JSON 解析异常。

**根因**：
- `parseResponse()` 方法用**字符串 indexOf** 手动解析 JSON（`json.indexOf("\"content\":\"")`），而不是用 Jackson
- 如果 content 里包含转义引号 `\"`、换行 `\n`、嵌套 JSON，这个简单搜索会找错位置
- `extractJsonFromResponse()` 在 `PptService` 中用 Jackson 解析，但 `chat()` 方法走的 `parseResponse()` 用手写解析 → **两套解析逻辑，一套有 bug**

**建议修复**：
- 统一使用 Jackson 解析 OpenAI 响应（和 `chatRaw` + `extractJsonFromResponse` 一样）

---

### 🔴 P1-04：文件拖拽上传 — PPT/Research 功能未实际对接

**现象**：
- PPT 页面拖拽文件后，AI 回复"传文件做参考的功能待完善" → **功能缺失**
- Research 页面拖拽文件后，只记录了文件名，**没有读取内容发给后端**
- DocQA 页面浏览器版只支持 txt/md，PDF/DOCX 根本不能读

**根因**：
- PPT `handleFile()` 只添加了 UI 消息，没有调任何后端 API
- Research `handleFile()` 同上
- DocQA 浏览器端无法解析 PDF，后端有 `DocParserService` 但浏览器无法直接使用

**建议修复**：
1. PPT/Research：前端把文件内容 base64 或 FormData 传给后端，后端解析后作为参考内容
2. DocQA 浏览器版：增加文件上传 API（multipart），后端解析后返回文本
3. 或者使用 File API + ArrayBuffer 读取后发送给后端解析

## 二、中等问题（功能不完整或体验差）

### 🟡 P2-01：PPT 大纲修改功能是假的

**现象**：用户在大纲阶段提出修改意见，AI 只回复"已记录您的调整意见"，但**实际上什么都没做**。

**根因**（PptGenerator.vue L114-116）：
```javascript
} else if (step.value === 1) {
    addMsg('ai', '好的，已记录您的调整意见。您可以继续修改，或点击「满意，生成PPT」')
}
```
没有把用户的修改意见发给后端重新生成大纲。

**建议修复**：将修改意见 + 原大纲一起发给后端，让 AI 修改大纲后返回新版本。

---

### 🟡 P2-02：Research 调研报告没有"下载"按钮

**现象**：生成报告后，后端返回了 `filePath` 和 `fileName`，但前端只显示了一个文字路径，没有下载链接。

**建议修复**：增加下载按钮，调用 `/api/research/download?file=xxx`（类似 PPT 的 download 接口）。

---

### 🟡 P2-03：Research 页面提示"联网检索文献" — 但实际没有联网

**现象**：前端提示"📚 我会联网检索相关文献"，但 DeepSeek API 本身**没有联网搜索能力**（除非用 DeepSeek-Search 模型），实际只是靠 AI 编造。

**根因**：
- 配置的 model 是 `deepseek-v4-flash`，这是普通对话模型
- 没有任何搜索 API 集成

**建议修复**：
1. 要么换用 DeepSeek-Search 模型（如果有的话）
2. 要么修改前端文案，去掉"联网检索"的说法
3. 长期方案：集成搜索 API（如 Serper、Bing Search）

---

### 🟡 P2-04：Session 数据全部存在内存，重启即丢失

**现象**：
- `DocQAController.sessions` 是 `ConcurrentHashMap`，重启后所有会话丢失
- 没有持久化存储

**影响**：文档问答的上下文在重启后丢失，用户需要重新上传。

**建议修复**：至少用本地文件（JSON/SQLite）持久化 session 数据。

---

### 🟡 P2-05：DocQA 的 `escape()` 方法有 bug

**根因**（DocQAController.java L138-144）：
```java
private String escape(String s) {
    return s.replace("\\", "\\\\")
            .replace("\"", "\\\"")
            .replace("\n", "\\n")
            .replace("\r", "\\r")
            .replace("\t", "\\t");
}
```
这个手动 escape 替代 JSON 序列化，顺序有问题：先替换 `\` 为 `\\`，再替换 `\n` 为 `\\n` → 结果变成 `\\\\n`（双转义）。

同样的问题也存在于 `DeepSeekService.escape()` 中。

**建议修复**：直接用 `ObjectMapper.writeValueAsString()` 做序列化，不要手写 escape。

---

### 🟡 P2-06：ChatHome 的通用对话没有上下文

**现象**：ChatHome 页面的对话是单轮的，每次发送都是独立请求，没有对话历史。

**根因**（ChatController.java）：
```java
String reply = deepSeek.chat(systemPrompt, message);
```
只传了当前消息，没有维护对话上下文。

**建议修复**：维护会话级别的消息历史，像 DocQA 一样传递上下文。

---

### 🟡 P2-07：PptService.generateFromScratch() 生成的 PPT 太简陋

**现象**：无模板时生成的 PPT 只有纯文本框，没有布局、配色、图标，基本不可用。

**根因**：`generateFromScratch()` 只用了 `createTextBox()` + `setText()`，没有任何样式设计。

**建议修复**：
1. 增加内置的默认模板（几种常见医学 PPT 风格）
2. 或者至少加背景色、字体大小、对齐方式等基本样式

---

### 🟡 P2-08：模型名称可能不存在

**现象**：配置文件中 `deepseek.model: deepseek-v4-flash`，但 DeepSeek 官方模型名称是 `deepseek-chat` 或 `deepseek-reasoner`，不存在 `deepseek-v4-flash`。

**建议修复**：确认实际可用的模型名称并修正配置。

---

## 三、轻度问题（代码质量/安全隐患）

### 🟢 P3-01：API Key 明文写在配置文件中

`application.yml` 中 `api-key` 通过环境变量注入，但代码中 `log.debug` 可能打印出 key。且没有验证 key 是否为空。

---

### 🟢 P3-02：前后端超时不匹配

| 层 | 超时 |
|---|---|
| 后端 HttpClient connectTimeout | 30s |
| 后端 HttpRequest timeout | 60s |
| 前端 axios timeout | 120s |

后端 60s 超时后前端还要等 60s 才报错，用户体验极差。

---

### 🟢 P3-03：所有 Controller 返回 `Map<String, Object>` 而非 `ApiResult<T>`

项目定义了 `ApiResult<T>` 统一响应类，但所有 Controller 都用 `Map.of()` / `HashMap` 返回，响应格式不统一。

---

### 🟢 P3-04：没有全局异常处理

没有 `@ControllerAdvice` + `@ExceptionHandler`，异常直接返回 Spring 默认的 500 Whitelabel 页面。

---

### 🟢 P3-05：PPT 下载接口有路径遍历风险

```java
String fileName = (String) body.getOrDefault("fileName", "");
File localFile = new File(System.getProperty("user.home") + "/yiyu/output", fileName);
```

`fileName` 如果传入 `../../etc/passwd`，可以读取系统任意文件。

---

### 🟢 P3-06：没有文件上传接口

前端拖拽文件后无法真正上传到后端，缺少 multipart upload API。

---

### 🟢 P3-07：DocQA 浏览器版 PDF/DOCX 无法使用

`DocQAView.vue` 的 `loadFile()` 方法只支持 txt/md/json 等文本文件，PDF/DOCX 直接提示"浏览器中当前仅支持"。

但 DocQA 产品定位是"上传 PDF 或 Word"，核心功能在浏览器版完全不可用。

---

### 🟢 P3-08：JavaFX Desktop 启动逻辑有死锁风险

`YiyuDesktop.init()` 在 JavaFX 线程同步启动 Spring Boot，`stop()` 在 JavaFX 线程调 `springContext.close()` 又套了 `Platform.runLater()`，可能导致死锁。

---

### 🟢 P3-09：pom.xml 重复依赖

`javafx-controls`、`javafx-web`、`javafx-media` 都同时引入了无 classifier 和 `win` classifier 版本，Mac 上不需要 win classifier。

---

### 🟢 P3-10：Tika 依赖引入但未使用

`pom.xml` 中引入了 `tika-core` 3.3.1，但代码中没有任何地方使用 Tika，文档解析用的是 PDFBox + POI。

---
